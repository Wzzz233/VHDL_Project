# OV5640 噪点优化进展（2026-03-10）

## 1. 背景
- 截至 `2026-03-10`，`main` 分支最新相关提交为 `058802c`。
- 在 `2026-03-09` 的 OV5640 初始化回归修复后，当前状态已经从“黑屏 / 雪花 / 花屏排查”进入“实时画面噪点优化”阶段。
- 目前用户常用运行命令仍以 `--fpga-preproc-profile raw` 为主，因此屏幕颗粒噪点主要仍由传感器自动增益与默认 CIP 行为决定，而不是由 FPGA 显示预处理决定。

## 2. 本轮改动

### 2.1 传感器侧
- 文件：`hdl/reg_config.v`
- 新增编译期噪点档位：
  - `OV5640_NOISE_PRESET_REF`
  - `OV5640_NOISE_PRESET_LOW_NOISE_BALANCED`
  - `OV5640_NOISE_PRESET_LOW_NOISE_DISPLAY`
- 当前默认档位：
  - `LOW_NOISE_BALANCED`
- 首轮只改两组寄存器：
  - 增益上限：`0x3a18/0x3a19`
  - CIP 锐化/降噪：`0x5300..0x530c`

### 2.2 显示侧
- 文件：`ARM/fpga_lpr_display.c`
- 新增运行时 profile：
  - `--fpga-preproc-profile median`
- 行为定义：
  - 只开 median
  - 不开 CLAHE
  - 不开 USM
  - 不开 OCR stroke 特殊路径
- 同步更新帮助文本：
  - `ARM/fpga_lpr_display.c`
  - `ARM/run_lpr_kms.sh`

## 3. 与需求的关系

### 3.1 对“图像预处理由 FPGA 实现”的支撑
对照 `Docs/最终项目要求.md` 中关于“图像灰度化、降噪、二值化等图像预处理”的要求：
- 当前项目已经具备 FPGA 侧预处理控制路径与多种 profile。
- 本轮新增的 `median` 进一步补强了“纯降噪、少增强”的测试能力。
- 但从交付角度看，这部分仍然是“部分完成”，因为还缺少完整的板级定量 A/B 结果来证明默认配置收益。

### 3.2 对“系统稳定运行”的支撑
- 当前代码状态下，OV5640 初始化回归已经修复，摄像头正常出图恢复。
- 本轮噪点优化没有引入新的运行时传感器 CLI，也没有扩大初始化链路修改范围，目的是在保持稳定出图的前提下做小步调参。

### 3.3 对“OCR/车牌识别”的影响
- 本轮默认优先级是“实时显示画面更干净”。
- 允许 OCR 细节略软，但不接受明显持续漏检。
- 因此本轮结论必须依赖板上 60s 连续运行结果，而不能只看单帧观感。

## 4. 当前建议测试矩阵

### 4.1 同一 bitstream 下的显示侧 A/B
- `raw`
- `median --fpga-preproc-target all`
- `median_clahe_usm --fpga-preproc-target all`

目的：
- 判断“仅显示侧滤波”是否足以明显降低颗粒噪点。

### 4.2 传感器位流 A/B
- `LOW_NOISE_BALANCED + raw`
- `LOW_NOISE_DISPLAY + raw`

目的：
- 判断“单靠 OV5640 传感器表”能否在不破坏稳定性的情况下改善画面。

## 5. 通过标准
- 屏幕无黑屏、雪花、条纹、错位回归。
- 室内/弱光场景下，细碎亮度颗粒相对历史基线明显下降。
- 颜色不过度发灰。
- 车牌与边缘允许略软，但不能出现明显持续识别恶化。
- 连续运行 60s 不应出现新的 DMA timeout 或程序异常退出。

## 6. 当前判断
- 当前代码已经提供了“传感器侧 + 显示侧”两条独立降噪杠杆。
- 最可能成为正式默认方案的是：
  - 若 `LOW_NOISE_BALANCED + raw` 已足够好，则直接采用该方案。
  - 若仍有明显颗粒，但 `median + target=all` 明显改善，则采用 `LOW_NOISE_BALANCED + median` 作为推荐运行组合。
- 只有在上述两者都不足时，才建议进入 `LOW_NOISE_DISPLAY` 的正式对比。
